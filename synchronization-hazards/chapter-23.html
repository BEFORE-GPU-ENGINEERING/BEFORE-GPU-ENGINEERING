What __syncthreads() actually protects

*the seam that holds cooperative GPU code together*

On a GPU, thousands of threads in a block run together. So if one thread writes a value into shared memory and another thread reads that same spot, the read should just get the right value. It doesn't, not reliably. And the reason why is one of the most counterintuitive truths in parallel programming: time is fluid across threads. Understanding this, and the single instruction that fixes it, is the difference between code that works and code that silently produces garbage.

The Illusion That Breaks Everything

Start with the illusion. We say threads run together, and loosely they do. But together is not the same as in perfect step with instant shared results. When one thread writes a value into shared memory, that write does not appear in the memory the same instant the instruction runs. The value has to travel through a pipeline of buffers before it physically lands in the memory cells and becomes visible to everyone else.

So if a second thread reads that spot a hair too early, while the write is still somewhere in the pipeline, it grabs the old value: a stale number, or plain garbage. This is called a read-after-write hazard. The reader arriving before the writer has truly finished.

↳ And here is the part that surprises even people who know GPUs. You might think that threads inside the same 32-wide warp, which genuinely do march instruction by instruction together, would be safe. They are not. Because the memory system is separate from the execution unit and runs on its own pipeline. Even as the warp steps forward in lockstep, a store instruction and a following load can both be in flight at once, and the load can reach the memory before the store has fully retired. Lockstep execution does not guarantee memory visibility.

The Barrier: Two Jobs, Not One

So you need a way to force everyone to stop and let the writes settle. That is the barrier. And the crucial thing is that it does two jobs, not one. It is easy to think of it as just a pause, everyone wait here. It is more than that.

First, it is a stop. When a thread hits the barrier, it freezes and waits until every single thread in the block has also reached that same line. The hardware keeps a count, and nobody moves past until the last straggler arrives.

Second, and this is the part people miss, it is a memory fence. Reaching the barrier forces all the pending writes still stuck in the pipeline to be flushed out and physically committed to memory. So the guarantee is not just "everyone arrived here." It is: everything everyone wrote before this line is now truly visible to everyone after this line.

↳ That combination, the stop plus the flush, is what makes the read safe. Everyone writes, everyone hits the barrier, the writes are guaranteed committed, and only then does anyone read.

The Trap That Hangs the Whole GPU

Now the trap that catches people constantly. It seems reasonable to think: if only some threads need to sync, let me guard the barrier inside an if. Say only the first half of the block wrote something, so you put the barrier inside the branch that only they take.

The GPU freezes. Dead. Because the barrier has one absolute rule: either every thread in the block reaches it, or none do.

Walk through why it hangs. The threads inside the branch arrive at the barrier and stop, waiting for the whole block to show up. But the threads that skipped the branch never reach that barrier at all. They moved on. So the waiting threads wait for teammates who are never coming. They wait forever. The barrier is counting for the entire block, and the count can never complete.

↳ Even splitting a barrier across both sides of an if-else doesn't save you, because a data-dependent branch means you can't guarantee the same threads take the same path. The only safe way is to put the barrier outside the branch entirely, on a line every thread is guaranteed to reach, whether it did the work or not. Everyone participates in the sync, even the ones who had nothing to write.

barrier inside an if + divergent threads = deadlock, forever.

Why the Barrier Is Scoped to a Block

This points at a deeper principle about why the barrier is scoped to a block and nothing larger. A block is the smallest unit of truly cooperative execution. The hardware gives you fast shared memory and this barrier within a block. But it gives you no barrier at all across different blocks in the same launch.

That is deliberate. If blocks could all wait on each other, the GPU would lose its freedom to run blocks in any order on whatever processing units happen to be free. That freedom is exactly what lets it scale. Barriers require tracking global state, and that gets expensive fast. So the hardware draws a hard line: cheap synchronization inside a block, none across blocks.

↳ And if you genuinely need to synchronize across the whole grid, the answer is to end the kernel and start a new one. The first kernel fully finishing before the second begins is itself the synchronization, with the results passed through memory in between.

Three Hazards, One Root

Worth naming that the barrier guards more than just read-after-write. There are three sibling hazards whenever threads touch the same shared spot:

  1. Read-after-write: A reader jumps the gun on a writer and grabs a stale value before the write has landed.
  2. Write-after-read: A writer overwrites a value before another thread has finished reading the old one.
  3. Write-after-write: Two writes to the same spot land in an unpredictable order, leaving whichever value the hardware chose, not whichever the programmer intended.

All three come from the same root: without a forced ordering, the hardware and compiler assume memory operations are independent and are free to reorder them. The barrier is what imposes the order.

↳ The General Takeaway

Threads on a GPU do not share a perfectly synchronized sense of time, and memory writes take a moment to become visible even within a single warp. So a naive read can grab stale data. The barrier fixes this by doing two things at once: halting every thread in the block until all arrive, and flushing all pending writes to memory so they are visible afterward.

But it comes with an iron rule. Every thread in the block must reach it, or the whole thing deadlocks forever. Which means it can never live inside a divergent branch. And it only works within a block, because cheap local synchronization is the price the GPU pays to keep blocks independent and scalable.

The barrier is a small thing to write...
but it is the seam that holds cooperative GPU code together.
