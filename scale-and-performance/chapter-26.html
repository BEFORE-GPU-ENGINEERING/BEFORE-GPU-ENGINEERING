Why the GPU context switch is free but CPU's is slow

The Free Swap

A CPU with 8 cores wants about 8 threads. A GPU with a few thousand cores wants hundreds of thousands of threads. That ratio makes no sense until you understand the one trick the whole GPU is built around, and the piece of hardware that makes the trick cost nothing. The GPU deliberately floods itself with far more work than it has cores, and then flips between that work for free. Understanding why that flip is free, when the same flip on a CPU is painfully expensive, explains the deepest design choice in the entire chip.

Start with the problem the GPU is solving. Fetching data from main memory takes a very long time, hundreds of clock cycles, because the memory sits physically far from the cores and the request has to travel out, fetch, and travel back. That is a fixed physical cost, no chip can make it disappear. So the question is what the cores do during that long wait. And the naive answer is a disaster. If you launch exactly one thread per core, then every core fires off its memory request at once, every core stalls waiting hundreds of cycles, and there is no other work to do. The cores sit frozen. All that expensive silicon, idle. Giving the GPU exactly as many threads as cores wastes almost the entire machine.

Overwhelming the Hardware

The escape is to overwhelm it on purpose. You launch not one batch of threads but dozens of batches per core, far more than can run at once. The threads run in fixed bundles of 32, and each processing unit keeps many such bundles resident at the same time. Now when one bundle fires its memory request and stalls, the scheduler instantly puts it to sleep and wakes another bundle whose data is already sitting ready. When that one stalls, it wakes the next. By the time the scheduler has cycled through enough bundles, the first one's data has finally arrived from memory, and it is ready to run again.

The cores never see an idle moment, because there is always another bundle with work ready to go. The long memory wait is still happening, you just hid it behind a wall of other work. This is the GPU's entire performance philosophy, it does not make any single thread fast, it makes sure the cores never wait by always having a backup army ready.

Why the Switch is Free

But this only works if switching from one bundle to the next is cheap. If every switch cost what a switch costs on a CPU, the cure would be worse than the disease. And here is the beautiful part, on a GPU the switch is essentially free, zero or one cycle, no data moved at all. To see why, you have to look at where the thread's working state, its registers, physically lives.

So first, what actually happens when a CPU switches from one thread to another. A CPU core has one small set of registers, a single scratch space built right into its execution pipeline for the one thread it is running. Those registers hold that thread's live values. When the operating system decides to switch to a different thread, the core cannot just abandon the first thread's values, they are precious. So it has to copy all of the first thread's register values out into memory, then fetch the second thread's saved values from memory and load them into that same small register space, and often flush some cached state too. That copying back and forth takes thousands of cycles and is managed by the operating system. A CPU context switch is genuinely heavy, real work moving real data.

Now the GPU, which made the opposite choice. Instead of one small register space shared by whoever is running, a GPU processing unit has one enormous register file, big enough to hold the complete live state of every resident bundle all at once, side by side. When a bundle is assigned to run on that unit, it is handed its own private slice of that giant register file, and it keeps that slice for its entire life. Bundle 0's registers, bundle 1's registers, bundle 2's, they all sit in physical silicon simultaneously, none of them overwriting the others.

So ask the key question. When the scheduler switches from one bundle to another, what data moves? The answer is none. Nothing is saved, nothing is loaded. Both bundles' full states are already sitting in the register file, permanently, the whole time. The switch is nothing more than the scheduler updating a pointer, telling the execution units to now read from a different slice of the register file. A pointer flip. That is why it costs zero or one cycle while the CPU's costs thousands. The CPU pays because it has to shuffle state in and out of a small shared space. The GPU pays nothing because it never shuffles, it just points somewhere else in a space large enough to hold everyone at once.

↳ The General Takeaway

This trade is the whole identity of the two chips. The CPU spends its silicon on making a single thread as fast as possible, with big caches and clever prediction, and accepts that switching threads is expensive because it does not switch very often.

The GPU spends its silicon on a gigantic register file so it can hold a huge crowd of threads resident and flip between them for free, and accepts that any single thread runs slowly because it is never trying to be fast at one thing, it is trying to keep every core fed. This is exactly why a lone thread with no parallel siblings runs poorly on a GPU. With nothing to switch to, the free-switch machinery is useless, and you are left with one slow thread on hardware that was never built to rush a single one.

The GPU's gift and its cost are the same fact.
It is a machine for hiding waiting, and it hides it by keeping thousands ready and switching between them at no cost at all.
